Human-centered evaluation Human-centered evaluation to ensure clarity and trust in AI outputs. PLANNED The Game Monitor with expandable board snapshots per move already enables human review. Adding "why did the AI choose this?" tooltips is a natural extension of the GreedyPlayer's score breakdown.
1.4 Data Science & Analytics
Feature engineering from gameplay data Feature engineering from gameplay data. DONE BasicProfileAnalyzer extracts 10+ features: color preferences, source/dest splits, row preferences, timing stats, score trajectory. Every move is stored with full board state — rich feature space for any model.
Clustering and classification for opponent archetypes Clustering and classification to derive opponent archetypes. PLANNED Data pipeline is ready (JSON export with all features). Adding a scikit-learn clustering analyzer is straightforward — register a new ProfileAnalyzer subclass. Architecture explicitly supports this.

Modular agent-of-agents architecture Design of a modular agent-of-agents architecture, where specialized agents handle rule extraction, move prediction, and profiling. DONE Implemented via ABC interfaces: MachinePlayer (move prediction), ProfilerAgent (profiling), ProfileAnalyzer (analysis). Registry pattern for pluggable components. Each agent is independent and composable.

7. Progress Update — System Design Components

Component Detail from Report Status Commentary
7.1 System Design
Game state representation Game state representation for Azul. DONE GameStateData Pydantic model: factories, center_pool, players (pattern_lines, wall, floor_line, score). JSON-serializable, stored in SQLite per move.
Rulebook ingestion Rulebook ingestion into validator. RECONSIDER Rules are hand-coded in azul/rules.py. NLP ingestion of a PDF rulebook is a research project in itself. Recommend: skip for Azul, demonstrate as a concept with a simple RAG demo on the side.
Validator component Deterministic rules + state transitions; the only component allowed to mutate game state. DONE azul/rules.py — get_legal_actions(), can_place_on_pattern_line(), score_tile_placement(), etc. 47 unit tests. The server (buddyboardgames.com) is the ultimate validator; our engine validates client-side before emitting.
Experiment tracking / Ablation study MLflow or similar for experiment tracking. PLANNED Not implemented. The JSON export format contains all data needed for analysis. Adding MLflow integration is moderate effort. Recommend: use the JSON export + Jupyter notebooks for Phase 2 analysis, add MLflow in Phase 3.
7.3 Multi-Agent Decision Stack
Validator (Authoritative) Deterministic rules + state transitions; the only component allowed to mutate game state. DONE See above. Also: the game platform server acts as the authoritative validator — our engine submits moves and checks success: true/false responses.
Profiler Produces a brief distribution over opponent archetypes and/or a predictive opponent policy. PARTIAL BasicProfileAnalyzer produces style classification and feature distributions. Not yet: archetype distribution (requires clustering), predictive opponent policy (requires sequence model).
Tactician Chooses actions via MCTS/Minimax (optionally guided by learned priors/value) conditioned on Profiler output. PARTIAL GreedyPlayer is a heuristic tactician (single-step lookahead). MCTS/Minimax would use the existing get_legal_actions() + score_tile_placement() as the search foundation. This is the highest-impact next step.
Explainer Renders a faithful explanation by consuming artefacts emitted by Validator/Tactician. PLANNED Not implemented as a separate component. The GreedyPlayer's score breakdown per action is the natural "artefact" for explanation. Adding an Explainer that converts score vectors to natural language is moderate effort.
7.4 Agentic Systems Design
LangGraph orchestration Graph/state-driven orchestration (LangGraph) and multi-agent conversation abstractions (AutoGen). RECONSIDER The current architecture uses simple Python async + Registry pattern. LangGraph/AutoGen add significant complexity for marginal benefit in a turn-based game. The agents don't need "conversation" — they need state-in, action-out. Recommend: keep the current clean architecture; mention LangGraph compatibility as a future direction.
7.5 Game State Representation for Azul
State variables (Factories, Tile Supply, Floor pool, Pattern lines, Wall, Floor line) All Azul state variables defined. DONE All implemented in GameStateData + PlayerState Pydantic models. Serialized as JSON, stored per move.
JSON or Embeddings representation Exploring JSON or embeddings for state representation. DONE JSON representation fully implemented and in production. Embeddings would be a model-specific preprocessing step — the JSON provides the raw data for any embedding scheme.
Action space: (source, color, destination) Each move could be (source, color, destination). DONE GameAction model: source_type (factory/center), source_index, color, destination (pattern_line/floor), destination_row. Exact match to the report's proposal.
7.6 Agent Communication Protocol
Latency and telemetry constraints We have latency and telemetry constraints. Telemetry could be a module. DONE Every move records decision_time_ms, click_ms, ws_wait_ms, total_ms. System Log broadcasts real-time timing for every phase. Configurable move_timeout_sec + stuck_abort_sec.
Traces of action moves for agents Traces of the action moves for agents. DONE Immutable move ledger in SQLite. Every move has action, board state snapshot, timing, player name, model tag. JSON export includes full trace. System log captures every phase.
Explanations / thinking / reason / profile update from the agent Agent-level explanations and reasoning traces. PLANNED Not yet emitted per-move. The GreedyPlayer's score breakdown per candidate action is available internally — exposing it requires adding a explain() method to the MachinePlayer ABC. Moderate effort.
Performance metrics (self-report collector) Self-report metrics collector installed per agent. DONE System log captures per-move: DOM parse time, ML decision time, click execution time, WS confirmation time. Profiler computes per-player statistics.
7.8 Profiler Design
Cold-start handling Initialize prior + Bayesian updating. High-uncertainty gating: down-weight opponent-conditioning in search. HIGH EFFORT Not implemented. Requires a probabilistic profiler model (not rule-based). The current BasicProfileAnalyzer works post-game. A real-time Bayesian profiler that updates during the game is a significant ML engineering effort. Recommend for Phase 3.
7.9 Tactician Design
MCTS/Minimax integration Integrated profiling into MCTS/Minimax. PLANNED The rules engine provides everything MCTS needs: get_legal_actions(), score_tile_placement(), is_game_over(). The full-game simulation in test_full_game.py already demonstrates state transitions. MCTS implementation is the next high-impact feature.
RL + search hybrid pathway RL + search hybrid. HIGH EFFORT Requires: (1) RL training pipeline, (2) value/policy network, (3) integration with MCTS. The game simulation infrastructure is ready for generating training data. Recommend: start with pure MCTS, add learned value function in Phase 3.
7.10 Explainability and Experimentation
Mapping RL/MCTS values to natural language Pipeline: Tactician emits structured factors, Validator emits semantic features, Explainer performs counterfactual comparison. PLANNED GreedyPlayer already computes per-action scores with named factors (wall_score, fill_ratio, overflow_penalty). Adding natural language mapping is moderate effort. Counterfactual comparison ("what if you picked red instead?") requires running the evaluator on alternative actions — straightforward with the rules engine.
Experiment system (MLflow) Build experiment tracking tools — MLflow. PLANNED Not started. The JSON export format serves as experiment data. MLflow can be added for tracking model performance across games. Low priority for MVP.
Isolate Profiler win-rate contribution Experiment: Isolate and measure the win-rate contribution of the Profiler vs baseline MCTS. PLANNED The tournament simulation infrastructure (test_full_game.py) can run GreedyPlayer-with-profiling vs. GreedyPlayer-without-profiling. This is a valuable experiment once MCTS + Profiler are both implemented.
